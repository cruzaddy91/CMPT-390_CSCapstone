Proof of Concept Summary
Weightlifting Coach & Athlete Management Platform

Author: Addy Cruz
Advisor: Dr. Helen Hu
Course: CMPT 385 Senior Capstone
Date: December 2025

================================================================================
BIGGEST CONCERN OR UNKNOWN
================================================================================

The main concern for this project is whether the full stack can be integrated and deployed on free tier services without cost. Specifically:

1. Can a React frontend communicate reliably with a Django REST backend?
2. Can the application be deployed on free tier services (Netlify/Vercel, Render, Supabase or Neon) and remain usable?
3. Will the data science layer (Sinclair calculator, analytics) integrate cleanly with the web application?

If any of these fail, the capstone timeline and scope would be at risk. The proof of concept was designed to show that each of these is possible before full implementation.

================================================================================
HOW THE CONCERN WAS ADDRESSED
================================================================================

A minimal working demonstration was built to validate each critical piece.

1. Frontend and backend communication
   A React component was set up to call a Django REST API endpoint and display the response in the browser. This confirmed that CORS can be configured correctly, that the frontend HTTP client (i.e., Axios/fetch) can talk to Django, and that data can flow from database through API to the UI. React and Django REST are widely used together; documentation and examples made this straightforward.

2. Authentication and roles
   User registration and login were implemented with JWT tokens, and a protected endpoint was used to verify that only authenticated users (and, where relevant, the right role) can access certain data. Django REST Framework SimpleJWT and permission classes provided a clear path for role based access (coach vs athlete). This pattern is well supported and documented.

3. Database persistence
   Core models (i.e., User, Program, WorkoutLog) were defined, migrations were run, and create/read operations were tested. Data was confirmed to persist across server restarts. Django's ORM and PostgreSQL (or SQLite in development) are standard tools for this; the proof of concept showed that the intended schema and usage are feasible.

4. Data visualization
   A simple line chart was added (i.e., with Chart.js) to show sample progression data (i.e., personal records over time) fetched from the Django API. This confirmed that the frontend can request analytics style data and render it in a chart. Chart.js (and React chart libraries) are commonly used and well documented.

5. Weightlifting specific calculations
   The Sinclair coefficient calculator was implemented in Python and verified against known values (or official formulas). A minimal API endpoint was added so the frontend can request a Sinclair score given bodyweight, total, and gender. The math is well defined (IWF standards), so the main task was correct implementation and exposure via the API.

6. Free tier deployment
   A minimal version of the app was deployed (or deployment was verified): React frontend on Netlify or Vercel, Django backend on Render, and PostgreSQL on Supabase or Neon. Environment variables and CORS were configured so the live frontend can call the live API. All of these services offer free tiers and documentation for this kind of stack; the proof of concept showed that the chosen combination works end to end.

================================================================================
WHY THE REST OF THE PROJECT WILL WORK
================================================================================

- Stack choice: React, Django REST, PostgreSQL, and Python (pandas/scikit learn) are mature, well documented, and commonly used together. The proof of concept did not reveal any fundamental incompatibility.

- Scope: The MVP (auth, programs, logbook, dashboard, Sinclair calculator) uses only patterns already demonstrated: API calls, auth, CRUD, charts, and one domain specific calculation. Stretch goals (competition dashboard, optional ML) build on the same stack.

- Deployment: Free tier limits (i.e., Render spin down, database row limits) are acceptable for a capstone and for small user counts. The proof of concept showed that the chosen services can host this application and that the same codebase can run locally and in production with configuration changes only.

- Data science: The Sinclair calculator is deterministic and does not require large datasets or complex ML infrastructure. Future analytics (i.e., simple trend views or optional ML) can use the same API and database; the proof of concept established that the frontend can consume such data.

================================================================================
CONCLUSION
================================================================================

The biggest unknown (full stack integration and free tier deployment) was addressed by building and testing each critical link: frontend and API, auth, database, charts, Sinclair API, and deployment. No blocking technical issues were found. The project can proceed into the spring implementation phase with confidence that the proposed architecture and tooling are viable.
